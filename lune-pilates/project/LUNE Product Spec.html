Screen-by-screen essentials

What must live on each home surface — the non-negotiable elements for each audience.

Customer Dashboard

Member & guest · mobile

  • Balance & expiry, hero-level
    Hours remaining + nearest expiry date, with a "household" marker when it's a shared pool.
  • Next class card
    Date, time, type; one tap to detail, reschedule or cancel.
  • One primary action — Book a class
    Everything else is secondary. No competing CTAs.
  • Policy clarity at point of action
    The 5-hour free-cancel cutoff is shown before booking and on every booking.
  • Household sharing surface
    Who's on the house number, shared balance, invite a member.
  • Buy credits / PromptPay
    Category-segmented catalog → QR checkout → instant balance update.

Admin Dashboard

Front desk · mobile-first, responsive to iPad & desktop

  • Today at a glance
    Classes, attendee counts, check-in status, utilisation — the opening screen.
  • Schedule mgmt — baseline + publish
    Edit a week from the template, set instructor & capacity, publish.
  • Bookings & waitlist control
    Manual book/cancel, and waitlist with the notify-and-confirm window.
  • Customers (not just members)
    House numbers, sharing groups, balances; add a new customer on the spot.
  • POS checkout
    Sell packages & retail, take PromptPay or cash, issue a receipt.
  • Instructor availability & Gantt
    Set who works when; a vertical day timeline shows lanes per instructor.
05

Logic & architecture

Two mechanisms carry most of the product's complexity: shared-credit deduction and tiered visibility.

House / family sharing

Model the package as belonging to a household, not a person. Members are linked to a household by house number; deductions draw from one shared balance via an append-only ledger.

Household
id · house_number · owner_user_id · created_at
User
id · phone · name · tier (member | guest) · household_id?
Package (balance holder)
id · type · hours_total · hours_left · expires_at
· owner: household_id (member) · OR user_id (guest)
CreditLedger (append-only)
id · package_id · delta (−1 / +5) · actor_user_id · booking_id? · reason · created_at
Deduction = one atomic transaction: re-check hours_left > 0 and expires_at > now, insert a −1 ledger row, decrement hours_left, write the booking — all or nothing. The ledger's actor_user_id answers "who used the household's hours," so sharing never becomes a dispute.
Guest packages set owner = user_id and skip the household join entirely — non-transferable by construction, no special-casing needed.

Tiered visibility (early booking)

Don't store two schedules. Store one class instance with two computed timestamps, then filter by the viewer's tier and the current time.

ClassInstance
id · template_id? · starts_at · type · capacity · instructor_id?
· status (draft | published)
· members_visible_at = published_at
· public_visible_at = starts_at − N hrs
The booking query, simplified
where status = 'published'
and starts_at > now()
and ( viewer.tier = 'member'
    or now() >= public_visible_at )
Member Sees every published class the moment the week goes live — books days ahead. Guest Sees a class only once now ≥ starts_at − N (e.g. 24 hrs before) — fills the late, open spots.
N is a single tunable. Want guests to see further out for under-filled classes? Lower N per class type, or derive it from current occupancy — no schema change, just the formula behind public_visible_at.
Waitlist follows the same discipline: joining writes a Waitlist row, never a booking. When a seat frees, the system offers it to the head of the queue with a 30-minute confirm window (a short-lived hold); on timeout it cascades to the next person. No auto-charge, no surprise booking.

Automation & CRM

LINE is where this audience already lives — make it the primary channel, event-driven off the same system actions.

TriggerChannel · messageTiming
Weekly schedule publishedLINE  Broadcast: next week is live, with a "Book now" link.On admin publish
Booking confirmedLINE  Flex card: class, time, instructor, add-to-calendar + balance left.Instant
Class reminderLINE  Gentle nudge with cancel/reschedule still in the free window.2 hrs before
Waitlist spot opensLINE  "A spot opened — confirm within 30 min to claim it." Countdown link.On seat free · 30-min hold
Credit expiry approachingLINE  Hours left + expiry date, one-tap re-purchase.7 days & 1 day before
Household balance lowLINE  To the household owner: pool is down to 1 hr — top up.At ≤ 1 hr left
Post-class follow-upLINE  Thank-you + quick rebook of the same slot next week.~2 hrs after class
Win-back (lapsing)LINE  Warm check-in for members inactive 30 days.Day 30 of inactivity
Every row is fired by a system event already in the data model (publish, booking insert, ledger change, waitlist offer) — so the CRM is a thin listener, not a parallel source of truth. SMS or push can mirror any row as a fallback when a member hasn't linked LINE.
